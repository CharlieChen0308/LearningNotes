# 05 反射與代理

> **版本**：Java 17+ — 涵蓋 Module System 對反射的限制、三種代理模式比較、Spring 的代理選擇策略

## 1、反射（Reflection）

反射允許在執行期間檢查和操作類別、方法、欄位，是 Spring IoC、MyBatis、JPA 等框架的底層基礎。

### 1.1 取得 Class 物件

```java
// 方式 1：類別字面量（編譯期確定，推薦）
Class<String> clazz1 = String.class;

// 方式 2：物件的 getClass()
String s = "hello";
Class<?> clazz2 = s.getClass();

// 方式 3：Class.forName()（執行期動態載入）
Class<?> clazz3 = Class.forName("java.lang.String");

// 三者取得的是同一個 Class 物件
System.out.println(clazz1 == clazz2);  // true
System.out.println(clazz2 == clazz3);  // true
```

### 1.2 Class.forName() vs ClassLoader.loadClass()

| 方法 | 觸發初始化 | 用途 |
|------|-----------|------|
| `Class.forName("X")` | 是（執行靜態區塊） | JDBC 載入驅動（`DriverManager` 在靜態區塊中註冊） |
| `ClassLoader.loadClass("X")` | 否（僅載入） | Spring 的 Lazy Loading |

```java
// JDBC 範例（Java 6+ 其實 SPI 自動載入，不需要手動寫）
Class.forName("com.mysql.cj.jdbc.Driver");  // 注意：不加 .class 後綴

// ClassLoader 範例
Class<?> clazz = Thread.currentThread()
    .getContextClassLoader()
    .loadClass("com.example.MyService");
```

### 1.3 反射操作

```java
public class User {
    private String name;
    private int age;

    public User() {}
    public User(String name, int age) { this.name = name; this.age = age; }

    private String greet() { return "Hello, " + name; }
}

// 建立物件
Class<User> clazz = User.class;
Constructor<User> ctor = clazz.getDeclaredConstructor(String.class, int.class);
User user = ctor.newInstance("Alice", 30);

// 存取 private 欄位
Field nameField = clazz.getDeclaredField("name");
nameField.setAccessible(true);  // 繞過 private 限制
String name = (String) nameField.get(user);  // "Alice"
nameField.set(user, "Bob");

// 呼叫 private 方法
Method greet = clazz.getDeclaredMethod("greet");
greet.setAccessible(true);
String result = (String) greet.invoke(user);  // "Hello, Bob"
```

### 1.4 Module System 的限制（Java 9+）

Java 9 引入模組系統後，`setAccessible(true)` 對其他模組的類別可能拋出 `InaccessibleObjectException`：

```
Unable to make field private final byte[] java.lang.String.value accessible:
module java.base does not "opens java.lang" to unnamed module
```

**解決方案**：
- JVM 啟動參數：`--add-opens java.base/java.lang=ALL-UNNAMED`
- 框架（如 Spring、Hibernate）已透過 `module-info.java` 或 JVM 參數處理

## 2、靜態代理

代理模式：為目標物件提供一個代理，控制對目標的存取。

```java
// 介面
public interface UserService {
    void save(String name);
}

// 目標類別
public class UserServiceImpl implements UserService {
    @Override
    public void save(String name) {
        System.out.println("Saving user: " + name);
    }
}

// 靜態代理
public class UserServiceProxy implements UserService {
    private final UserService target;

    public UserServiceProxy(UserService target) {
        this.target = target;
    }

    @Override
    public void save(String name) {
        System.out.println("[LOG] Before save");
        target.save(name);
        System.out.println("[LOG] After save");
    }
}
```

**缺點**：每個介面都要寫一個代理類別，程式碼膨脹嚴重。

## 3、JDK 動態代理

基於 `java.lang.reflect.Proxy`，**只能代理介面**：

```java
UserService target = new UserServiceImpl();

UserService proxy = (UserService) Proxy.newProxyInstance(
    target.getClass().getClassLoader(),
    target.getClass().getInterfaces(),
    (proxyObj, method, args) -> {
        System.out.println("[LOG] Before " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("[LOG] After " + method.getName());
        return result;
    }
);

proxy.save("Alice");
// [LOG] Before save
// Saving user: Alice
// [LOG] After save
```

## 4、CGLIB 動態代理

基於位元組碼生成，**可以代理沒有介面的類別**：

```java
// CGLIB（Spring 內建）
public class CglibExample {
    public static void main(String[] args) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(UserServiceImpl.class);
        enhancer.setCallback((MethodInterceptor) (obj, method, arguments, proxy) -> {
            System.out.println("[LOG] Before " + method.getName());
            Object result = proxy.invokeSuper(obj, arguments);
            System.out.println("[LOG] After " + method.getName());
            return result;
        });

        UserServiceImpl proxy = (UserServiceImpl) enhancer.create();
        proxy.save("Alice");
    }
}
```

**限制**：不能代理 `final` 類別和 `final` 方法（因為 CGLIB 透過繼承實現）。

## 5、三種代理比較

| 特性 | 靜態代理 | JDK 動態代理 | CGLIB 代理 |
|------|---------|-------------|-----------|
| 實現方式 | 手寫代理類別 | `java.lang.reflect.Proxy` | 位元組碼生成（繼承） |
| 前提條件 | 實作相同介面 | 目標必須有介面 | 不需要介面 |
| 效能 | 最快（編譯期確定） | 反射呼叫，較慢 | 方法呼叫快，建立慢 |
| `final` 限制 | 無 | 無 | 不能代理 final 類別/方法 |
| 維護成本 | 高（每個介面一個代理） | 低 | 低 |

## 6、Spring 的代理選擇策略

Spring AOP 預設行為（Spring Boot 2.0+）：

```
1. 如果目標類別有介面 → 仍使用 CGLIB（Spring Boot 2.0+ 改變了預設值）
2. spring.aop.proxy-target-class=true（預設） → CGLIB
3. spring.aop.proxy-target-class=false → 有介面用 JDK，無介面用 CGLIB
```

> **Spring Boot 2.0+ 預設使用 CGLIB**，因為 CGLIB 代理避免了「注入介面但呼叫的方法不在介面上」的問題。

### 為何 Spring 需要代理？

Spring 的 `@Transactional`、`@Async`、`@Cacheable` 等註解都是透過 AOP 代理實現的。當你呼叫一個帶有這些註解的方法時，實際上呼叫的是代理物件的方法，代理在方法前後加入切面邏輯。

這也是為什麼**同類別內部呼叫不生效** — 因為內部呼叫不經過代理。

## 7、小結

| 概念 | 重點 |
|------|------|
| 反射 | 執行期操作類別/方法/欄位；框架的基礎 |
| Class.forName | 載入並初始化類別（觸發靜態區塊） |
| JDK 動態代理 | 基於介面，Proxy.newProxyInstance |
| CGLIB | 基於繼承，不需要介面；不能代理 final |
| Spring 選擇 | Boot 2.0+ 預設 CGLIB |
| Module System | Java 9+ 限制跨模組反射存取 |
