# 03 物件導向進階

> **版本**：Java 17+ — 涵蓋介面 `default` 方法、`sealed` 類別、`record` 等現代特性

## 1、類別的例項化順序

面試常見題目：建立子類別物件時，各區塊的執行順序為何？

```java
public class Parent {
    static { System.out.println("1. Parent 靜態區塊"); }
    { System.out.println("4. Parent 實例區塊"); }
    public Parent() { System.out.println("5. Parent 建構子"); }
}

public class Child extends Parent {
    static { System.out.println("2. Child 靜態區塊"); }
    { System.out.println("6. Child 實例區塊"); }
    public Child() { System.out.println("7. Child 建構子"); }

    public static void main(String[] args) {
        System.out.println("3. main 方法開始");
        new Child();
        System.out.println("--- 第二次建立 ---");
        new Child();
    }
}
```

**輸出**：

```
1. Parent 靜態區塊
2. Child 靜態區塊
3. main 方法開始
4. Parent 實例區塊
5. Parent 建構子
6. Child 實例區塊
7. Child 建構子
--- 第二次建立 ---
4. Parent 實例區塊
5. Parent 建構子
6. Child 實例區塊
7. Child 建構子
```

**規則**：
1. **靜態區塊**：類別載入時執行一次，父類別先於子類別
2. **實例區塊 + 建構子**：每次 `new` 都執行，父類別先於子類別
3. 靜態區塊只執行一次（第二次 `new` 不再執行）

## 2、抽象類別 vs 介面

### 2.1 比較表

| 特性 | 抽象類別 | 介面 |
|------|---------|------|
| 關鍵字 | `abstract class` | `interface` |
| 欄位 | 可有實例欄位 | 只能有 `static final` 常量 |
| 建構子 | 可以有 | 不能有 |
| 方法 | 抽象 + 具體方法 | 抽象 + `default` + `static` + `private`（Java 9+） |
| 繼承 | 單繼承 | 多實作 |
| 存取修飾 | 任意 | 方法預設 `public` |

### 2.2 介面的 default 方法（Java 8+）

```java
public interface Loggable {
    // 抽象方法
    String getName();

    // default 方法：有預設實作，子類別可覆寫
    default void log(String message) {
        System.out.println("[" + getName() + "] " + message);
    }

    // static 方法（Java 8+）
    static Loggable of(String name) {
        return () -> name;  // 用 Lambda 實作 SAM 介面
    }

    // private 方法（Java 9+）：提取 default 方法中的共用邏輯
    private String formatMessage(String msg) {
        return "[" + java.time.LocalDateTime.now() + "] " + msg;
    }
}
```

### 2.3 何時用抽象類別 vs 介面

- **介面**：定義「能力」（can-do），如 `Serializable`、`Comparable`、`Runnable`
- **抽象類別**：定義「是什麼」（is-a），需要共享狀態或建構邏輯

> 現代 Java 傾向優先使用介面 + `default` 方法。只有在需要共享狀態（實例欄位）時才用抽象類別。

## 3、繼承 vs 組合

### 3.1 繼承的問題

```java
// 脆弱基底類別問題
public class InstrumentedHashSet<E> extends HashSet<E> {
    private int addCount = 0;

    @Override
    public boolean add(E e) {
        addCount++;
        return super.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return super.addAll(c);  // 內部呼叫 add()，導致重複計數！
    }
}
```

### 3.2 組合優於繼承（Effective Java Item 18）

```java
// 用組合 + 委託取代繼承
public class InstrumentedSet<E> implements Set<E> {
    private final Set<E> delegate;
    private int addCount = 0;

    public InstrumentedSet(Set<E> delegate) {
        this.delegate = delegate;
    }

    @Override
    public boolean add(E e) {
        addCount++;
        return delegate.add(e);
    }

    @Override
    public boolean addAll(Collection<? extends E> c) {
        addCount += c.size();
        return delegate.addAll(c);  // 不再有重複計數問題
    }

    public int getAddCount() { return addCount; }

    // 其餘 Set 方法全部委託給 delegate...
    @Override public int size() { return delegate.size(); }
    @Override public boolean isEmpty() { return delegate.isEmpty(); }
    @Override public boolean contains(Object o) { return delegate.contains(o); }
    @Override public Iterator<E> iterator() { return delegate.iterator(); }
    @Override public Object[] toArray() { return delegate.toArray(); }
    @Override public <T> T[] toArray(T[] a) { return delegate.toArray(a); }
    @Override public boolean remove(Object o) { return delegate.remove(o); }
    @Override public boolean containsAll(Collection<?> c) { return delegate.containsAll(c); }
    @Override public boolean retainAll(Collection<?> c) { return delegate.retainAll(c); }
    @Override public boolean removeAll(Collection<?> c) { return delegate.removeAll(c); }
    @Override public void clear() { delegate.clear(); }
}
```

### 3.3 選擇原則

| 情境 | 選擇 |
|------|------|
| A **是** B（is-a） | 繼承，但要確保 LSP |
| A **有** B（has-a） | 組合 |
| 需要多型 | 介面 + 組合 |
| 不確定 | **優先組合** |

## 4、equals() 和 hashCode()

### 4.1 equals() 合約

```java
public class Money {
    private final int amount;
    private final String currency;

    public Money(int amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;
        if (!(o instanceof Money money)) return false;  // pattern matching (Java 16+)
        return amount == money.amount && currency.equals(money.currency);
    }

    @Override
    public int hashCode() {
        return Objects.hash(amount, currency);
    }
}
```

**規則**：覆寫 `equals()` 就**必須**同時覆寫 `hashCode()`，否則 HashMap/HashSet 行為異常。

> 使用 `record` 可以自動獲得正確的 `equals()` 和 `hashCode()`：
> ```java
> public record Money(int amount, String currency) {}
> ```

## 5、Sealed Classes（Java 17）

限制類別的繼承層次，搭配 pattern matching 實現窮舉式 switch：

```java
public sealed interface Payment permits CreditCard, BankTransfer, Cash {}

public record CreditCard(String number, int cvv) implements Payment {}
public record BankTransfer(String iban) implements Payment {}
public record Cash(int amount) implements Payment {}

// 處理所有可能的子型別（編譯器檢查窮舉性）
public static String process(Payment payment) {
    return switch (payment) {
        case CreditCard cc -> "信用卡: " + cc.number();
        case BankTransfer bt -> "匯款: " + bt.iban();
        case Cash c -> "現金: " + c.amount() + " 元";
        // 不需要 default — 編譯器知道所有可能的型別
    };
}
```

## 6、小結

| 概念 | 要點 |
|------|------|
| 例項化順序 | 父靜態 → 子靜態 → 父實例+建構 → 子實例+建構 |
| 介面 vs 抽象類別 | 介面定義能力（多實作）；抽象類別共享狀態（單繼承） |
| 組合優於繼承 | 避免脆弱基底類別問題，增加彈性 |
| equals/hashCode | 必須一起覆寫；用 `record` 自動產生 |
| sealed | Java 17，限制繼承 + 編譯器窮舉檢查 |
