# 06 常用關鍵字與設計模式

> **版本**：Java 17+ — 涵蓋 `effectively final`、枚舉單例、反序列化攻擊防護

## 1、Java 的參數傳遞：值傳遞

Java **只有值傳遞**（pass by value），沒有引用傳遞：

```java
// 基本型別：傳遞值的副本
public static void changeValue(int x) {
    x = 100;  // 只修改副本，不影響原變數
}

int a = 42;
changeValue(a);
System.out.println(a);  // 42（不變）

// 參考型別：傳遞引用的副本（引用本身是值）
public static void changeName(User user) {
    user.setName("Bob");  // 透過引用副本修改物件 → 影響原物件
}

public static void replaceUser(User user) {
    user = new User("Charlie");  // 讓引用副本指向新物件 → 不影響原引用
}

User u = new User("Alice");
changeName(u);
System.out.println(u.getName());  // "Bob"（物件被修改）

replaceUser(u);
System.out.println(u.getName());  // "Bob"（原引用不變）
```

**記憶口訣**：
- 基本型別：傳遞值的**拷貝**
- 參考型別：傳遞引用的**拷貝**（可以透過拷貝修改物件，但不能讓原引用指向新物件）

## 2、final 關鍵字

### 2.1 三種用途

| 用途 | 效果 |
|------|------|
| `final` 變數 | 只能賦值一次（常量） |
| `final` 方法 | 不能被子類別覆寫 |
| `final` 類別 | 不能被繼承（如 `String`、`Integer`） |

### 2.2 final 變數的細節

```java
// 1. 編譯期常量
static final int MAX = 100;  // 會被內聯（inline）

// 2. 實例 final 欄位：必須在建構子結束前賦值
class Config {
    final String name;
    Config(String name) { this.name = name; }
}

// 3. final 參考只保證「引用不變」，物件內容仍可修改
final List<String> list = new ArrayList<>();
list.add("hello");  // OK：修改物件內容
// list = new ArrayList<>();  // 編譯錯誤：不能重新指向

// 4. 不可變集合
final var immutable = List.of("a", "b");
// immutable.add("c");  // UnsupportedOperationException
```

### 2.3 effectively final（Java 8+）

沒有 `final` 關鍵字，但從未被重新賦值的變數，Lambda 和匿名類別可以捕獲：

```java
int count = 10;  // effectively final
Runnable r = () -> System.out.println(count);  // OK

int mutable = 10;
mutable = 20;  // 重新賦值 → 不再 effectively final
// Runnable r2 = () -> System.out.println(mutable);  // 編譯錯誤
```

## 3、堆（Heap）與棧（Stack）

| 區域 | 存放內容 | 執行緒 | 生命週期 |
|------|---------|--------|---------|
| 棧（Stack） | 區域變數、方法呼叫幀、基本型別值 | 每個執行緒私有 | 方法結束即釋放 |
| 堆（Heap） | 物件實例、陣列 | 所有執行緒共享 | GC 回收 |

```java
public void example() {
    int x = 10;            // x 在棧上，值 10 也在棧上
    String s = "hello";    // s（引用）在棧上，"hello" 物件在堆上（字串常量池）
    User u = new User();   // u（引用）在棧上，User 物件在堆上
}
// 方法結束：x, s, u 從棧彈出
// User 物件如果沒有其他引用，等待 GC 回收
```

## 4、單例模式

### 4.1 五種實現方式比較

| 方式 | 執行緒安全 | 延遲載入 | 防反射攻擊 | 防反序列化攻擊 |
|------|-----------|---------|-----------|--------------|
| 餓漢式 | 是 | 否 | 否 | 否 |
| 懶漢式（synchronized） | 是 | 是 | 否 | 否 |
| 雙重檢查鎖 | 是 | 是 | 否 | 否 |
| 靜態內部類 | 是 | 是 | 否 | 否 |
| **枚舉**（推薦） | 是 | 否 | **是** | **是** |

### 4.2 餓漢式

```java
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private Singleton() {}
    public static Singleton getInstance() { return INSTANCE; }
}
```

類別載入時就建立實例。簡單但無延遲載入。

### 4.3 雙重檢查鎖（DCL）

```java
public class Singleton {
    private static volatile Singleton instance;  // volatile 防止指令重排

    private Singleton() {}

    public static Singleton getInstance() {
        if (instance == null) {                  // 第一次檢查（無鎖）
            synchronized (Singleton.class) {
                if (instance == null) {           // 第二次檢查（有鎖）
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

**為何需要 `volatile`**：`new Singleton()` 並非原子操作，JVM 可能重排序為：
1. 分配記憶體
2. 將引用指向記憶體（此時 instance != null）
3. 執行建構子

如果另一個執行緒在步驟 2 之後、步驟 3 之前讀取 instance，會取得未初始化的物件。

### 4.4 靜態內部類

```java
public class Singleton {
    private Singleton() {}

    private static class Holder {
        static final Singleton INSTANCE = new Singleton();
    }

    public static Singleton getInstance() {
        return Holder.INSTANCE;  // 首次呼叫時才載入 Holder 類別
    }
}
```

利用 JVM 的類別載入機制保證執行緒安全，且延遲載入。

### 4.5 枚舉單例（推薦）

```java
public enum Singleton {
    INSTANCE;

    private final Map<String, String> cache = new ConcurrentHashMap<>();

    public void doSomething() {
        // 業務邏輯
    }
}

// 使用
Singleton.INSTANCE.doSomething();
```

**為何枚舉是最佳方案**（Effective Java Item 3）：
- **執行緒安全**：JVM 保證枚舉實例只建立一次
- **防反射攻擊**：`Constructor.newInstance()` 對枚舉會拋出 `IllegalArgumentException`
- **防反序列化攻擊**：JVM 特殊處理枚舉的序列化，不會建立新實例
- **程式碼最簡潔**

### 4.6 反射攻擊防護

對於非枚舉的單例，可以加入防護：

```java
public class Singleton {
    private static final Singleton INSTANCE = new Singleton();
    private static boolean created = false;

    private Singleton() {
        if (created) {
            throw new IllegalStateException("Singleton already created!");
        }
        created = true;
    }

    // 防反序列化攻擊
    private Object readResolve() {
        return INSTANCE;
    }
}
```

## 5、其他常見設計模式速查

| 模式 | 一句話說明 | Java/Spring 實例 |
|------|-----------|-----------------|
| 工廠方法 | 定義建立物件的介面，由子類別決定實例化哪個類別 | `Calendar.getInstance()` |
| 抽象工廠 | 建立一系列相關物件 | `DocumentBuilderFactory` |
| 建造者 | 分步建構複雜物件 | `StringBuilder`、Lombok `@Builder` |
| 策略 | 封裝可互換的演算法 | `Comparator`、Spring 的 `AuthenticationProvider` |
| 觀察者 | 一對多通知 | Spring `ApplicationEvent` |
| 模板方法 | 定義演算法骨架，子類別填充步驟 | `JdbcTemplate`、`RestTemplate` |
| 代理 | 控制對物件的存取 | Spring AOP、JDK/CGLIB 動態代理 |
| 裝飾者 | 動態增加功能 | `BufferedReader(new FileReader(...))` |

## 6、小結

| 概念 | 重點 |
|------|------|
| 參數傳遞 | Java 只有值傳遞；參考型別傳遞引用的拷貝 |
| final | 變數不可重新賦值、方法不可覆寫、類別不可繼承 |
| effectively final | Java 8+，Lambda 可捕獲未重新賦值的變數 |
| 堆 vs 棧 | 物件在堆、引用在棧；堆共享、棧私有 |
| 單例模式 | 枚舉最佳；DCL 需要 volatile；靜態內部類也推薦 |
