# 03 交易與鎖機制

> **版本**：PostgreSQL 15+ / MySQL 8.0+ (InnoDB)

---

## 1、ACID 特性

ACID 是關聯式資料庫交易的四大保證：

| 特性 | 英文 | 說明 | 實現機制 |
|-----|------|------|---------|
| 原子性 | Atomicity | 交易中的操作全部成功或全部回滾 | Undo Log |
| 一致性 | Consistency | 交易前後資料滿足所有約束條件 | 應用層 + 約束 |
| 隔離性 | Isolation | 併發交易互不干擾 | MVCC + 鎖 |
| 持久性 | Durability | 提交後的資料永久保存，即使系統崩潰 | WAL / Redo Log |

```sql
-- 交易基本語法
BEGIN;  -- 或 START TRANSACTION;
    UPDATE accounts SET balance = balance - 1000 WHERE id = 1;
    UPDATE accounts SET balance = balance + 1000 WHERE id = 2;
COMMIT; -- 成功則提交

-- 發生錯誤時
ROLLBACK; -- 全部回滾
```

**Savepoint（儲存點）**：允許部分回滾

```sql
BEGIN;
    INSERT INTO orders (customer_id, total) VALUES (1, 5000);
    SAVEPOINT sp1;
    INSERT INTO order_items (order_id, product_id) VALUES (1, 999); -- 假設失敗
    ROLLBACK TO SAVEPOINT sp1;  -- 只回滾到 sp1，orders 的 INSERT 保留
    INSERT INTO order_items (order_id, product_id) VALUES (1, 100); -- 重試
COMMIT;
```

## 2、隔離級別

### 併發問題

| 問題 | 說明 | 範例 |
|-----|------|------|
| 髒讀 (Dirty Read) | 讀到其他交易未提交的資料 | TX-A 修改但未提交，TX-B 讀到修改值，TX-A 回滾 |
| 不可重複讀 (Non-Repeatable Read) | 同一交易內兩次讀取結果不同 | TX-B 讀取，TX-A 提交修改，TX-B 再讀取值已變 |
| 幻讀 (Phantom Read) | 同一條件查詢，行數增減 | TX-B 查詢 10 筆，TX-A 插入 1 筆提交，TX-B 再查變 11 筆 |

### 四大隔離級別

| 隔離級別 | 髒讀 | 不可重複讀 | 幻讀 | 效能 |
|---------|------|----------|------|------|
| READ UNCOMMITTED | 可能 | 可能 | 可能 | 最高 |
| READ COMMITTED | 防止 | 可能 | 可能 | 高 |
| REPEATABLE READ | 防止 | 防止 | 可能* | 中 |
| SERIALIZABLE | 防止 | 防止 | 防止 | 最低 |

### PostgreSQL vs MySQL 預設差異

| 特性 | PostgreSQL | MySQL InnoDB |
|-----|-----------|-------------|
| 預設隔離級別 | READ COMMITTED | REPEATABLE READ |
| REPEATABLE READ 防幻讀 | 透過 Snapshot Isolation 防止 | 透過 Next-Key Lock 防止 |
| READ UNCOMMITTED 行為 | 實際等同 READ COMMITTED（不允許髒讀） | 真正的 READ UNCOMMITTED |
| SERIALIZABLE 實現 | SSI (Serializable Snapshot Isolation) | 所有 SELECT 自動加共享鎖 |

*MySQL InnoDB 在 REPEATABLE READ 下透過 Next-Key Lock 已能防止大部分幻讀場景。

### 設定隔離級別

```sql
-- PostgreSQL
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 或全局設定
ALTER SYSTEM SET default_transaction_isolation = 'read committed';

-- MySQL
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
-- 或全局
SET GLOBAL transaction_isolation = 'REPEATABLE-READ';
```

## 3、MVCC 機制

MVCC（Multi-Version Concurrency Control）是讀寫不互相阻塞的關鍵機制。讀操作看到的是資料的某個歷史快照，寫操作不會阻塞讀操作。

### PostgreSQL MVCC：Tuple Versioning

PostgreSQL 在每個 tuple（資料行）上維護版本資訊：

| 系統欄位 | 說明 |
|---------|------|
| `xmin` | 插入此 tuple 的交易 ID |
| `xmax` | 刪除/更新此 tuple 的交易 ID（0 表示有效） |

```sql
-- 查看 tuple 的版本資訊
SELECT xmin, xmax, * FROM orders WHERE id = 1;
```

**UPDATE 操作的過程**：
1. 將舊 tuple 標記為已刪除（設定 xmax）
2. 插入一個新 tuple（新的 xmin）
3. 舊 tuple 由 VACUUM 程序回收

**可見性判斷規則**（簡化版）：
- `xmin` 已提交 且 `xmax` 未設定 → 可見
- `xmin` 已提交 且 `xmax` 已提交 → 不可見（已被刪除/更新）
- `xmin` 未提交 → 不可見（其他交易還沒提交的資料）

**VACUUM 的重要性**：
```sql
-- 手動執行 VACUUM
VACUUM VERBOSE orders;

-- 查看死元組數量
SELECT relname, n_dead_tup, n_live_tup, last_vacuum
FROM pg_stat_user_tables WHERE relname = 'orders';
```

PostgreSQL 的 autovacuum 會自動回收，但高寫入量的表可能需要調整 autovacuum 參數。

### MySQL InnoDB MVCC：Undo Log + Read View

InnoDB 使用不同的機制實現 MVCC：

**Undo Log**：
- 每次 UPDATE/DELETE 都在 undo log 記錄舊版本
- 透過 roll pointer 串成版本鏈
- Purge 執行緒定期清理不再需要的 undo log

**Read View**（讀取視圖）：
- 交易開始時建立一個 Read View，記錄當前活躍的交易 ID 列表
- 讀取時沿版本鏈查找第一個可見版本

| 隔離級別 | Read View 建立時機 |
|---------|------------------|
| READ COMMITTED | 每條 SELECT 都建立新的 Read View |
| REPEATABLE READ | 交易第一條 SELECT 時建立，整個交易復用 |

### 比較表

| 特性 | PostgreSQL | MySQL InnoDB |
|-----|-----------|-------------|
| 舊版本存放 | 原 Table 中（標記 xmax） | Undo Log（獨立儲存空間） |
| 版本清理 | VACUUM（autovacuum） | Purge 執行緒 |
| UPDATE 成本 | 較高（新增 tuple + 索引更新） | 較低（undo log + 聚簇索引原地更新） |
| 清理不及時的影響 | 表膨脹（table bloat） | Undo tablespace 膨脹 |
| 讀取長交易影響 | 阻止 VACUUM 回收，表持續膨脹 | 阻止 purge 清理，undo log 堆積 |

## 4、鎖機制

### PostgreSQL 鎖

**行級鎖**：

```sql
-- 排他鎖：FOR UPDATE（寫操作）
SELECT * FROM orders WHERE id = 1 FOR UPDATE;

-- 共享鎖：FOR SHARE（只讀，但阻止其他交易修改）
SELECT * FROM orders WHERE id = 1 FOR SHARE;

-- NOWAIT：無法取得鎖時立即報錯，不等待
SELECT * FROM orders WHERE id = 1 FOR UPDATE NOWAIT;

-- SKIP LOCKED：跳過已被鎖定的行（適用於任務佇列）
SELECT * FROM tasks WHERE status = 'PENDING'
ORDER BY created_at LIMIT 1
FOR UPDATE SKIP LOCKED;
```

**表級鎖**：

| 鎖模式 | 觸發操作 | 衝突 |
|-------|---------|------|
| ACCESS SHARE | SELECT | 只與 ACCESS EXCLUSIVE 衝突 |
| ROW SHARE | SELECT FOR UPDATE/SHARE | 與 EXCLUSIVE 衝突 |
| ROW EXCLUSIVE | INSERT/UPDATE/DELETE | 與 SHARE 以上衝突 |
| ACCESS EXCLUSIVE | ALTER TABLE / DROP | 與所有鎖衝突 |

**Advisory Lock（諮詢鎖）**：應用層自定義的鎖，不綁定特定資料行。

```sql
-- 取得諮詢鎖（以訂單處理為例，鎖定特定客戶）
SELECT pg_advisory_lock(customer_id) FROM customers WHERE id = 42;

-- 嘗試取得，不等待
SELECT pg_try_advisory_lock(42);  -- 回傳 true/false

-- 釋放
SELECT pg_advisory_unlock(42);
```

### MySQL InnoDB 鎖

**Record Lock（記錄鎖）**：鎖定索引記錄。

```sql
-- 鎖定 id=1 的記錄
SELECT * FROM orders WHERE id = 1 FOR UPDATE;  -- 排他記錄鎖
```

**Gap Lock（間隙鎖）**：鎖定索引記錄之間的「間隙」，防止其他交易插入。

```sql
-- 假設 id 有 1, 5, 10
SELECT * FROM orders WHERE id BETWEEN 5 AND 10 FOR UPDATE;
-- Gap Lock 會鎖定 (5, 10) 之間的間隙，防止插入 id=6, 7, 8, 9
```

**Next-Key Lock（臨鍵鎖）**：Record Lock + Gap Lock 的組合，InnoDB 在 REPEATABLE READ 下的預設鎖行為。

```sql
-- Next-Key Lock 範例：鎖定記錄本身 + 前面的間隙
-- 假設索引值有 10, 15, 20
SELECT * FROM orders WHERE id = 15 FOR UPDATE;
-- 鎖定範圍：(10, 15]（包含 15，鎖住 10~15 之間的間隙）
```

| 鎖類型 | 作用 | 隔離級別 |
|-------|------|---------|
| Record Lock | 鎖定具體索引記錄 | 所有級別 |
| Gap Lock | 鎖定索引間隙，防止插入 | REPEATABLE READ |
| Next-Key Lock | Record + Gap，防幻讀 | REPEATABLE READ |
| Insert Intention Lock | 插入意向鎖，相容其他插入 | 所有級別 |

## 5、死鎖

### 死鎖成因

兩個以上的交易互相等待對方持有的鎖，形成循環等待：

```
TX-A: 鎖定 row 1 → 等待 row 2 的鎖
TX-B: 鎖定 row 2 → 等待 row 1 的鎖
```

典型場景：

```sql
-- TX-A
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- 鎖定 id=1
UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- 等待 id=2

-- TX-B（同時）
BEGIN;
UPDATE accounts SET balance = balance - 50 WHERE id = 2;   -- 鎖定 id=2
UPDATE accounts SET balance = balance + 50 WHERE id = 1;   -- 等待 id=1 → 死鎖
```

### 排查方法

**PostgreSQL**：

```sql
-- 查看當前鎖
SELECT pid, locktype, relation::regclass, mode, granted, waitstart
FROM pg_locks
WHERE NOT granted;

-- 查看等待中的查詢
SELECT blocked.pid AS blocked_pid,
       blocked_activity.query AS blocked_query,
       blocking.pid AS blocking_pid,
       blocking_activity.query AS blocking_query
FROM pg_catalog.pg_locks blocked
JOIN pg_catalog.pg_stat_activity blocked_activity ON blocked.pid = blocked_activity.pid
JOIN pg_catalog.pg_locks blocking ON blocking.locktype = blocked.locktype
    AND blocking.relation = blocked.relation
    AND blocking.pid != blocked.pid
JOIN pg_catalog.pg_stat_activity blocking_activity ON blocking.pid = blocking_activity.pid
WHERE NOT blocked.granted;

-- 終止造成死鎖的 session
SELECT pg_terminate_backend(pid);
```

**MySQL**：

```sql
-- 查看 InnoDB 鎖狀態
SHOW ENGINE INNODB STATUS\G

-- 查看等待中的鎖（performance_schema）
SELECT * FROM performance_schema.data_lock_waits;

-- 查看當前持有的鎖
SELECT * FROM performance_schema.data_locks;

-- 終止造成問題的連線
KILL <thread_id>;
```

### 預防策略

| 策略 | 說明 |
|-----|------|
| 固定鎖定順序 | 所有交易按相同順序存取資源（如：按 id 升冪） |
| 縮短交易時間 | 減少鎖持有時間，避免交易中做 I/O 或外部呼叫 |
| 使用 NOWAIT | 無法取得鎖時立即失敗，而非無限等待 |
| 降低隔離級別 | READ COMMITTED 比 REPEATABLE READ 鎖更少 |
| 合理索引 | 確保 UPDATE/DELETE 走索引，避免鎖定大量不必要的行 |
| 設定鎖等待超時 | `lock_timeout`（PostgreSQL）/ `innodb_lock_wait_timeout`（MySQL） |

```sql
-- PostgreSQL：設定鎖等待超時（毫秒）
SET lock_timeout = '5s';

-- MySQL：設定 InnoDB 鎖等待超時（秒）
SET innodb_lock_wait_timeout = 10;
```

## 6、Spring @Transactional 與 DB 隔離級別的關係

### Spring 如何設定隔離級別

```java
@Service
public class OrderService {

    // 使用 DB 預設隔離級別
    @Transactional
    public void createOrder(OrderDTO dto) { /* ... */ }

    // 明確指定隔離級別
    @Transactional(isolation = Isolation.READ_COMMITTED)
    public void updateOrder(Long orderId, OrderDTO dto) { /* ... */ }

    // 唯讀交易（DB 可針對此做最佳化）
    @Transactional(readOnly = true)
    public OrderDTO getOrder(Long orderId) { /* ... */ }

    // 設定超時（秒）
    @Transactional(timeout = 10)
    public void batchProcess() { /* ... */ }
}
```

### Spring 到 DB 的對應

Spring `@Transactional` 開始交易時，底層會執行：

```sql
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;  -- 依設定值
BEGIN;
-- ... 業務 SQL ...
COMMIT; -- 或 ROLLBACK
```

| Spring Isolation | 對應 SQL |
|-----------------|---------|
| `DEFAULT` | 不設定，使用 DB 預設（PG: READ COMMITTED, MySQL: REPEATABLE READ） |
| `READ_UNCOMMITTED` | `SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED` |
| `READ_COMMITTED` | `SET TRANSACTION ISOLATION LEVEL READ COMMITTED` |
| `REPEATABLE_READ` | `SET TRANSACTION ISOLATION LEVEL REPEATABLE READ` |
| `SERIALIZABLE` | `SET TRANSACTION ISOLATION LEVEL SERIALIZABLE` |

### 傳播行為（Propagation）

```java
// REQUIRED（預設）：有交易就加入，沒有就新建
@Transactional(propagation = Propagation.REQUIRED)

// REQUIRES_NEW：無論如何都新建交易（暫停外層交易）
@Transactional(propagation = Propagation.REQUIRES_NEW)

// NESTED：巢狀交易（用 SAVEPOINT 實現）
@Transactional(propagation = Propagation.NESTED)
```

### 常見陷阱

**陷阱 1：自呼叫（Self-Invocation）不生效**

```java
@Service
public class OrderService {
    public void process() {
        this.createOrder();  // @Transactional 不生效！因為繞過了 AOP Proxy
    }

    @Transactional
    public void createOrder() { /* ... */ }
}
```

修復方式：注入自身或拆分到不同的 Service。

**陷阱 2：Exception 類型不觸發回滾**

```java
// 預設只有 RuntimeException 和 Error 才回滾
// checked Exception 不回滾！
@Transactional
public void process() throws IOException {
    // ... IOException 不會觸發回滾
}

// 修復：明確指定回滾條件
@Transactional(rollbackFor = Exception.class)
public void process() throws IOException { /* ... */ }
```

**陷阱 3：長交易持有鎖**

```java
@Transactional
public void processOrder(Long orderId) {
    Order order = orderRepo.findById(orderId);  // 開始交易
    externalApi.notify(order);                   // 外部呼叫可能很慢！
    order.setStatus("NOTIFIED");
    orderRepo.save(order);                       // 直到這裡才提交
}
// 修復：將外部呼叫移到交易外
```

## 7、小結

| 要點 | 說明 |
|-----|------|
| ACID | 原子性(Undo) + 一致性(約束) + 隔離性(MVCC+鎖) + 持久性(WAL/Redo) |
| 隔離級別 | PG 預設 READ COMMITTED；MySQL 預設 REPEATABLE READ |
| MVCC | PG 用 tuple versioning（需 VACUUM）；MySQL 用 undo log（需 purge） |
| 鎖 | PG 提供 advisory lock、SKIP LOCKED；MySQL 有 Gap Lock 防幻讀 |
| 死鎖 | 固定存取順序 + 縮短交易 + 設定超時 |
| Spring | 注意 self-invocation、checked exception、長交易陷阱 |
