## 1、覓藍

### 1、資料庫索引，複合索引，唯一索引，哪些地方用到？

**建立索引：**
```
格式：
create index 索引名 on 表名 (列名);
例子：
create index index_name on users (name);
```

**複合索引：**

- 單一索引是指索引列為一列的情況，即新建索引的語句只實施在一列上;     
- 使用者可以在多個列上建立索引，這種索引叫做**複合索引(組合索引)**;     
- 複合索引在資料庫操作期間所需的開銷更小，可以代替多個單一索引;

> 參考：[複合索引的優點和注意事項](https://www.cnblogs.com/summer0space/p/7247778.html)

**唯一索引：**

Unique Index 就是額外新增的唯一性約束。該約束嚴格的保證索引列的取值唯一性。例如：使用者郵箱、使用者的手機號需要唯一索引

**聚簇索引：**

聚簇索引的索引順序與資料庫的物理儲存順序一致，並且一個表只能有唯一的一條聚簇索引。

> 參考：[和剛入門的菜鳥們聊聊--什麼是聚簇索引與非聚簇索引](https://www.cnblogs.com/auxg/p/Cluster-and-NonCluster-index.html)

**非聚簇索引：**

非聚簇索引的索引順序與資料庫的物理儲存順序沒有必然聯絡，一個表可以有多個非聚簇索引。

**用到索引的地方：**

* 根據商品的分類來查詢商品時，在商品表裡面的商品分類欄位建立索引
* 根據商品的釋出時間來查詢商品
* ~~根據訂單的狀態來查詢訂單時~~（在出現大量重複值的欄位建立索引將不會有什麼幫助；相反，還有可能降低資料庫的效能）
* 根據文章的分類來查詢文章時，在文章表裡面的文章分類欄位建立索引
* 根據文章的釋出時間來查詢文章時

**用到複合索引的地方：**

- 使用者登入時，在資料庫中查詢賬號和密碼，用到複合索引。
- 同時根據商品的分類和商品的釋出時間來查詢商品
- 同時根據文章的分類和文章的釋出時間來查詢文章

### 2、資料庫 char，varchar 的區別

char 的長度是固定的。而 varchar 的長度是自動調整的，也就是說，定義一個 char(10) 和 varchar(10)，如果存進去的是 ‘csdn’，那麼 char 所佔的長度依然為 10，除了字元 ‘csdn’ 外，後面跟六個空格，而 varchar 就立馬把長度變為 4 了，取資料的時候，char 型別的要用 trim() 去掉多餘的空格，而  varchar 是不需要的，儘管如此，char 的存取速度還是要比 varchar 要快得多，因為其長度固定，方便程式的儲存與查詢；但是 char 也為此付出的是空間的代價，因為其長度固定，所以難免會有多餘的空格佔位符佔據空間，可謂是以空間換取時間效率，而 varchar 是以空間效率為首位的。

例如：  
- 儲存使用者的手機號，用 char(11)  
- 儲存使用者的真實姓名，用 char(10)  
- 郵政編碼，用 char(6)

### 3、各種資料庫引擎的區別

- ISAM：查詢速度快
- MyISAM：在 ISAM 的基礎上增加了索引功能
- InnoDB：支援事務、支援外來鍵。查詢速度比前面兩個引擎慢

在使用 MySQL 的時候，你所面對的每一個挑戰幾乎都源於 ISAM 和 MyISAM 資料庫引擎不支援事務處理（transaction process）也不支援外來鍵。儘管要比 ISAM 和  MyISAM 引擎慢很多，但是InnoDB包括了對事務處理和外來鍵的支援，這兩點都是前兩個引擎所沒有的。

**MyISAM與InnoDB的區別**

InnoDB 和 MyISAM 基本的差別為：MyISAM 型別的表強調的是效能，其執行速度比 InnoDB 型別更快，但是不支援事務、外來鍵等功能，而 InnoDB 支援事務、外來鍵等功能。

**MyISAM適合：**

- 做很多 count 的計算
- 查詢非常頻繁、沒有事務
- 例如：使用者表、文章表、文章分類表

**InnoDB適合：**

- 要求事務、外來鍵的地方
- 例如：訂單表，訂單的建立和訂單的取消，需要用到事務

### 10、秒殺用 Redis 佇列？原子性

假設有10件庫存，就往佇列中push10個數。搶購開始後，每到來一個使用者，就從佇列中pop一個數，表示使用者搶購成功。當列表為空時，表示已經被搶光了。因為 Redis 列表的pop操作是原子的，即使有很多使用者同時到達，也是依次執行的。

上面的會導致一個使用者搶多個，思路：   
需要一個排隊佇列(比如：queue:1,以user_id為值的列表)和搶購結果佇列(比如：order:1,以user_id為值的列表)及庫存佇列(比如上面的goods_store:1)。高併發情況，先將使用者進入排隊佇列，用一個執行緒迴圈處理從排隊佇列取出一個使用者，判斷使用者是否已在搶購結果佇列，如果在則已搶購，否則未搶購，接著執行庫存減1，寫入資料庫，將此user_id使用者同時也進入結果佇列。


>參考：[redis實現高併發下的搶購/秒殺功能](https://www.cnblogs.com/phpper/p/7085663.html)

### 1、索引是什麼？主鍵和唯一索引的區別？索引的缺點？

索引就相當於一本書的目錄，可以加快資料庫的查詢速度。

一張表只能有一個主鍵，但可以有多個唯一索引。

缺點：

- 索引在加快查詢速度的同時也讓插入和更新資料變慢，因為插入和更新資料時要修改索引結構
- 索引使用不當，有可能讓速度變慢。例如：資料量很少的表就不用建立索引；
- 重複資料較多的欄位就不應該建立索引，否則不僅沒有加快查詢速度，反而有可能讓速度變慢。

### 3、php 實現一個雙向佇列

```php
class Deque {  
    private $queue=array();  
      
    function addFirst($item){ //頭入隊  
        return array_unshift($this->queue, $item);  
    }    
    function removeFirst(){ //頭出隊  
        return array_shift($this->queue);  
    }  
    function addLast($item){ //尾入隊  
        return array_push($this->queue, $item);  
    }  
    function removeLast(){ //尾出隊  
        return array_pop($this->queue);  
    }  
    function show(){ //顯示  
        echo implode(" ", $this->queue);  
    }  
    function clear(){ //清空  
        unset($this->queue);  
    }  
    function getFirst(){  
        return array_shift($this->queue);  
    }  
    function getLast(){  
        return array_pop($this->queue);  
    }  
    function getLength(){  
        return count($this->queue);  
    }  
} 
```

### 5、訊息佇列 RabbitMQ 的相關概念，初始化配置

* Broker：簡單來說就是訊息佇列伺服器實體。
* Exchange：訊息交換機，它指定訊息按什麼規則，路由到哪個佇列。
* Queue：訊息佇列載體，每個訊息都會被投入到一個或多個佇列。
* Binding：繫結，它的作用就是把 exchange 和 queue 按照路由規則繫結起來。
* Routing Key：路由關鍵字，exchange 根據這個關鍵字進行訊息投遞。
* vhost：虛擬主機，一個 broker 裡可以開設多個 vhost，用作不同使用者的許可權分離。
* producer：訊息生產者，就是投遞訊息的程式。
* consumer：訊息消費者，就是接受訊息的程式。
* channel：訊息通道，在客戶端的每個連線裡，可建立多個 channel，每個 channel 代表一個會話任務。

訊息佇列的使用過程大概如下：
1. 客戶端連線到 Broker 訊息佇列伺服器，開啟一個 channel。
2. 客戶端宣告一個exchange，並設定相關屬性。
3. 客戶端宣告一個queue，並設定相關屬性。
4. 在exchange和queue之間建立好繫結關係。
5. 客戶端投遞訊息到exchange。
6. exchange接收到訊息後，就根據訊息的key和已經設定好的繫結，進行訊息路由。

訊息佇列的持久化包括：
1. exchange持久化，在宣告時指定durable => 1
2. queue持久化，在宣告時指定durable => 1
3. 訊息持久化，在投遞時指定delivery_mode => 2（1是非持久化）

如果exchange和queue都是持久化的，那麼它們之間的繫結也是持久化的。如果exchange和queue兩者之間有一個持久化，一個非持久化，就不允許建立繫結。

### 8、Redis 秒殺 ，商品庫存佇列用一個 String 存一個數字，透過數字的遞減來減少庫存

思路：查詢 Redis 中是否有儲存庫存，如果有就直接更新 Redis。如果 Redis 沒有，才去資料庫中取出庫存並快取到 Redis 中。

```php
public function buy($goods_id = 0){
	if(!$goods_id){
		die("商品不存在！");
	}
	$redis = new Redis();
	$redis->connect('127.0.0.1', 6379);
	$stock = 0;
	if(!$redis->get("gid".$goods_id)){
		$stock = get_stock($goods_id); // 從資料庫獲取實際庫存
		$redis->set("gid".$goods_id, $stock);
	}else{
		$stock = $redis->get("gid".$goods_id);
	}

	if($stock > 0){
		//邏輯操作（生成訂單等操作）
		//coding here...
		
		// 更新庫存
		$redis->set("gid".$goods_id, $stock-1);
	}else{
		die("已賣完！");
	}
}
```

>參考：[使用 redis 減少 秒殺庫存 超賣思路](https://blog.csdn.net/wujiangwei567/article/details/51210600)

對於Redis而言，命令的原子性指的是：一個操作的不可以再分，操作要麼執行，要麼不執行。

Redis的操作之所以是原子性的，是因為Redis是單執行緒的。

### 9、Redis 的 List 資料型別是連結串列結構？

在Redis中，List型別是按照插入順序排序的字串連結串列。和資料結構中的普通連結串列一樣，我們可以在其頭部(left)和尾部(right)新增新的元素。在插入時，如果該鍵並不存在，Redis將為該鍵建立一個新的連結串列。與此相反，如果連結串列中所有的元素均被移除，那麼該鍵也將會被從資料庫中刪除。

從元素插入和刪除的效率視角來看，如果我們是在連結串列的兩頭插入或刪除元素，這將會是非常高效的操作，即使連結串列中已經儲存了百萬條記錄，該操作也可以在常量時間內完成。然而需要說明的是，如果元素插入或刪除操作是作用於連結串列中間，那將會是非常低效的。

### 12、資料庫引擎 MyISAM 為什麼比 InnoDB 快？MyISAM 的儲存結構？

- ISAM：查詢速度快
- MyISAM：在 ISAM 的基礎上增加了索引功能
- InnoDB：支援事務、支援外來鍵。查詢速度比前面兩個引擎慢

**MyISAM 為什麼比 InnoDB 快？**

1. InnoDB 要快取整個資料塊，MyISAM 只快取索引塊
2. InnoDB 定址要對映到資料塊，再到行。MyISAM 記錄的直接是檔案的 offset，定位比較快
3. InnoDB 還需要維護 MVCC 一致。MVCC (Multi-Version Concurrency Control)多版本併發控制

### 13、多程序之間通訊的幾種方式

- 管道
  - 它是半雙工的（即資料只能在一個方向上流動），具有固定的讀端和寫端
  - 它只能用於具有親緣關係的程序之間的通訊（也是父子程序或者兄弟程序之間）
  - 它可以看成是一種特殊的檔案，對於它的讀寫也可以使用普通的read、write 等函式。但是它不是普通的檔案，並不屬於其他任何檔案系統，並且只存在於記憶體中
- FIFO（也稱為命名管道，它是一種檔案型別）
  - FIFO可以在無關的程序之間交換資料
  - FIFO有路徑名與之相關聯，它以一種特殊裝置檔案形式存在於檔案系統中
- 訊息佇列
  - 訊息佇列是面向記錄的，其中的訊息具有特定的格式以及特定的優先順序
  - 訊息佇列獨立於傳送與接收程序。程序終止時，訊息佇列及其內容並不會被刪除
  - 訊息佇列可以實現訊息的隨機查詢，訊息不一定要以先進先出的次序讀取,也可以按訊息的型別讀取
- 訊號量（semaphore）：與前面介紹過的 IPC 結構不同，它是一個計數器。訊號量用於實現程序間的互斥與同步，而不是用於儲存程序間通訊資料。
  - 訊號量用於程序間同步，若要在程序間傳遞資料需要結合共享記憶體
  - 訊號量基於作業系統的 PV 操作，程式對訊號量的操作都是原子操作
  - 每次對訊號量的 PV 操作不僅限於對訊號量值加 1 或減 1，而且可以加減任意正整數
  - 支援訊號量組
- 共享記憶體（Shared Memory）：指兩個或多個程序共享一個給定的儲存區
  - 共享記憶體是最快的一種 IPC，因為程序是直接對記憶體進行存取
  - 因為多個程序可以同時操作，所以需要進行同步（或加鎖）
  - 訊號量+共享記憶體通常結合在一起使用，訊號量用來同步對共享記憶體的訪問
- Socket

> 參考：[程序間的五種通訊方式介紹](https://blog.csdn.net/wh_sjc/article/details/70283843)

### 20、MySQL 的正規化和反正規化

- 第一正規化（1NF）：資料庫表中的列具有原子性，不能再分解。只要資料庫是關係型資料庫(mysql/oracle/db2/informix/sysbase/sql server)，就自動的滿足1NF。
- 第二正規化（2NF）：要求資料庫表中的每一行具備唯一性。通常需要一個主鍵來區分它的唯一性
- 第三正規化（3NF）：要求一張表中不包含已在其它表中的非主鍵欄位。例如商品和商品分類應該設計成兩張表，而不是放在一張表裡面

反正規化：   
沒有冗餘的資料庫未必是最好的資料庫，有時為了提高執行效率，就必須降低正規化標準，適當保留冗餘資料。降低正規化就是增加欄位，減少了查詢時的關聯，提高查詢效率。 

>參考：[MySQL中資料中設計中的正規化與反正規化](https://www.cnblogs.com/olinux/p/5191062.html)

### 23、分庫分表

**垂直拆分**：將不同的表拆分到不同的機器上
- 優點：拆分規則明確、資料維護簡單
- 缺點：無法透過 join 連線查詢、存在單庫大資料

**水平拆分**：將同一張表的資料拆分到不同的機器上
- 優點：可以用 join 連線查詢、不存在單庫大資料
- 缺點：拆分規則難以抽象、資料維護比較難
- 水平拆分的規則：
  - 根據 id 值取餘，將餘數不同的資料分散到不同的機器上
  - 根據特定的範圍段分散到不同的庫中
  - 將不同日期的資料分散到不同的機器上

>參考：
