# 設計模式實戰應用

> **版本**：Java 17+ / Spring Boot 3.x — GoF 設計模式在企業開發的實戰應用

---

## 1、為什麼要學設計模式

設計模式不是拿來背的。如果你面試時能把 23 種 GoF 模式的定義倒背如流，卻在實際開發中寫出一堆 `if-else` 嵌套，那等於沒學。

設計模式真正的價值在於：**知道何時該用、何時不該用**。

### 不是背定義，是學「何時用、何時不用」

工作兩三年後，你一定遇過這樣的情境：

- 需求一變，好幾個類別要同時修改 → 缺少**策略模式**隔離變化點
- 一個方法裡面塞了五六個 `if` 分支 → 缺少**工廠模式**或**策略模式**解耦
- 想加快取、加日誌，結果到處 copy-paste → 缺少**裝飾者模式**或 **AOP**

設計模式就是前人踩坑後總結出來的「套路」，讓你不用重新發明輪子。

### 過度設計 vs 適度設計的判斷標準

一個簡單的判斷原則：

| 該用模式 | 不該用模式 |
|---------|-----------|
| 同一類邏輯已經出現 3 次以上的分支 | 功能明確、短期不會變動 |
| 需求方明確說「未來會擴充」 | 只有兩種情況，且不太可能增加 |
| 團隊多人協作，需要統一介面 | 個人工具腳本、一次性任務 |

記住一句話：**先讓它能跑，再讓它好看，最後才讓它有彈性**。

## 2、創建型模式

### 2.1 Factory Method — Spring 的 BeanFactory 就是工廠

**核心思想**：把物件建立的邏輯集中管理，呼叫方不需要知道具體實作類別。

#### 實際場景：根據支付類型建立不同處理器

```java
// 定義支付處理器介面
public interface PaymentProcessor {
    PaymentResult process(PaymentRequest request);
    String getType(); // "CREDIT_CARD", "LINE_PAY", "BANK_TRANSFER"
}

// 工廠類別
@Component
public class PaymentProcessorFactory {

    private final Map<String, PaymentProcessor> processorMap;

    // Spring 會自動注入所有 PaymentProcessor 實作
    public PaymentProcessorFactory(List<PaymentProcessor> processors) {
        this.processorMap = processors.stream()
                .collect(Collectors.toMap(PaymentProcessor::getType, Function.identity()));
    }

    public PaymentProcessor getProcessor(String paymentType) {
        PaymentProcessor processor = processorMap.get(paymentType);
        if (processor == null) {
            throw new UnsupportedPaymentTypeException(paymentType);
        }
        return processor;
    }
}
```

這裡的關鍵是 **Spring 的依賴注入天然支援工廠模式**。你只要實作介面、加上 `@Component`，Spring 就會幫你收集所有實作。

#### Spring 實例：FactoryBean

Spring 本身的 `FactoryBean<T>` 就是工廠模式的體現。當你需要建立一個初始化流程複雜的 Bean 時：

```java
@Component
public class RestClientFactoryBean implements FactoryBean<RestClient> {

    @Override
    public RestClient getObject() {
        return RestClient.builder()
                .baseUrl("https://api.example.com")
                .defaultHeader("X-API-Key", "secret")
                .requestInterceptor(new LoggingInterceptor())
                .build();
    }

    @Override
    public Class<?> getObjectType() {
        return RestClient.class;
    }
}
```

### 2.2 Builder — 複雜物件的建構

**核心思想**：當建構子參數超過 4 個，或者有多種可選組合時，用 Builder 提升可讀性。

#### Lombok @Builder vs 手寫 Builder

日常開發中，90% 的場景用 Lombok 就夠了：

```java
@Builder
@Getter
public class OrderSearchCriteria {
    private String customerName;
    private LocalDate startDate;
    private LocalDate endDate;
    private OrderStatus status;
    private Integer minAmount;
    private Integer maxAmount;
    @Builder.Default
    private int page = 0;
    @Builder.Default
    private int size = 20;
}

// 使用
var criteria = OrderSearchCriteria.builder()
        .customerName("王大明")
        .startDate(LocalDate.of(2026, 1, 1))
        .status(OrderStatus.COMPLETED)
        .build();
```

**什麼時候需要手寫 Builder？** 當你需要建構過程中的驗證邏輯，例如「結束日期不能早於開始日期」。這時候在 `build()` 方法裡加入驗證，比在建構子裡更清晰。

### 2.3 Singleton — Spring Bean 預設就是 Singleton

#### 為什麼 Spring 選擇 Singleton Scope

Spring 容器中的 Bean 預設是 Singleton，原因很實際：

1. **效能**：不用每次請求都建立新物件
2. **狀態共享**：Service、Repository 這類無狀態元件，天生適合共用
3. **資源管理**：資料庫連線池、HTTP Client 這些重量級資源，不可能每次 new 一個

```java
@Service  // 預設就是 Singleton
public class OrderService {

    private final OrderRepository orderRepository;

    // 透過建構子注入，每次注入的都是同一個實例
    public OrderService(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }
}
```

#### 注意：Singleton + 可變狀態 = 執行緒安全問題

這是新手最常踩的坑：

```java
@Service
public class PriceCalculator {
    // 千萬不要這樣做！Singleton 的成員變數會被所有執行緒共享
    private BigDecimal totalAmount = BigDecimal.ZERO;

    public void addItem(BigDecimal price) {
        totalAmount = totalAmount.add(price); // 執行緒不安全！
    }
}
```

**正確做法**：Singleton Bean 只持有無狀態的依賴（其他 Service、Repository），計算用的臨時狀態放在方法的區域變數裡。

## 3、結構型模式

### 3.1 Decorator — 功能層層疊加

**核心思想**：在不修改原始物件的前提下，動態地為它增加新功能。

#### Java I/O Stream 就是 Decorator

你每天都在用裝飾者模式：

```java
// BufferedReader 裝飾了 InputStreamReader，InputStreamReader 裝飾了 FileInputStream
var reader = new BufferedReader(
    new InputStreamReader(
        new FileInputStream("data.csv"), StandardCharsets.UTF_8));
```

每一層都只負責一件事：`FileInputStream` 讀位元組、`InputStreamReader` 轉字元編碼、`BufferedReader` 加緩衝。

#### 實際場景：為 Service 加上快取裝飾

```java
public interface ProductService {
    Product findById(Long id);
}

@Service
@Primary
public class CachedProductService implements ProductService {

    private final ProductService delegate;
    private final CacheManager cacheManager;

    public CachedProductService(
            @Qualifier("productServiceImpl") ProductService delegate,
            CacheManager cacheManager) {
        this.delegate = delegate;
        this.cacheManager = cacheManager;
    }

    @Override
    public Product findById(Long id) {
        Cache cache = cacheManager.getCache("products");
        Product cached = cache.get(id, Product.class);
        if (cached != null) {
            return cached;
        }
        Product product = delegate.findById(id);
        cache.put(id, product);
        return product;
    }
}
```

當然，Spring 的 `@Cacheable` 已經幫你做了這件事。但理解背後的裝飾者模式，能讓你在 `@Cacheable` 不夠用時（例如需要條件性快取、多層快取），自己實作。

### 3.2 Proxy — Spring AOP 的核心

**核心思想**：為目標物件提供一個替身，在不改動目標物件的情況下攔截對它的存取。

#### JDK Dynamic Proxy vs CGLIB

| 特性 | JDK Dynamic Proxy | CGLIB |
|------|-------------------|-------|
| 前提 | 目標類別必須實作介面 | 不需要介面 |
| 原理 | 基於 `java.lang.reflect.Proxy` | 基於位元組碼生成子類別 |
| Spring Boot 3.x 預設 | 否 | **是**（`proxyTargetClass=true`） |
| 效能 | 反射呼叫，稍慢 | 直接方法呼叫，較快 |

#### 實際場景：@Transactional 就是 Proxy

```java
@Service
public class OrderService {

    @Transactional
    public void createOrder(OrderRequest request) {
        // Spring 實際上建立了一個 Proxy 包裹這個方法
        // Proxy 做的事：
        // 1. 開啟交易 (beginTransaction)
        // 2. 呼叫你的 createOrder 邏輯
        // 3. 正常結束 → commit
        // 4. 拋出例外 → rollback
        orderRepository.save(toEntity(request));
        inventoryService.deduct(request.getItems());
    }
}
```

**常見陷阱**：同一個類別中的方法互相呼叫，`@Transactional` 不會生效，因為呼叫沒有經過 Proxy。

```java
@Service
public class OrderService {

    public void processOrder(OrderRequest request) {
        // 這裡直接呼叫 this.createOrder()，不經過 Proxy
        // @Transactional 不會生效！
        createOrder(request);
    }

    @Transactional
    public void createOrder(OrderRequest request) { /* ... */ }
}
```

## 4、行為型模式

### 4.1 Strategy — 最實用的模式

這是日常開發中使用頻率最高的模式，沒有之一。

#### 錯誤：if-else 地獄

```java
// 每次新增支付方式，都要改這個方法 → 違反開放封閉原則
public PaymentResult pay(String type, PaymentRequest request) {
    if ("CREDIT_CARD".equals(type)) {
        // 30 行信用卡邏輯
    } else if ("LINE_PAY".equals(type)) {
        // 25 行 LINE Pay 邏輯
    } else if ("BANK_TRANSFER".equals(type)) {
        // 20 行銀行轉帳邏輯
    }
    // 下個月要加 Apple Pay... 繼續加 else if？
}
```

#### 正確：Strategy + Spring 注入

```java
// 策略介面
public interface PaymentStrategy {
    PaymentResult pay(PaymentRequest request);
    String getType();
}

// 信用卡策略
@Component
public class CreditCardStrategy implements PaymentStrategy {

    @Override
    public PaymentResult pay(PaymentRequest request) {
        // 信用卡專屬邏輯：呼叫金流 API、3D 驗證等
        return PaymentResult.success("信用卡付款完成");
    }

    @Override
    public String getType() { return "CREDIT_CARD"; }
}

// LINE Pay 策略
@Component
public class LinePayStrategy implements PaymentStrategy {

    @Override
    public PaymentResult pay(PaymentRequest request) {
        // LINE Pay 專屬邏輯：產生付款連結、回呼處理等
        return PaymentResult.success("LINE Pay 付款完成");
    }

    @Override
    public String getType() { return "LINE_PAY"; }
}

// 使用端 — 乾淨俐落
@Service
public class PaymentService {

    private final Map<String, PaymentStrategy> strategyMap;

    public PaymentService(List<PaymentStrategy> strategies) {
        this.strategyMap = strategies.stream()
                .collect(Collectors.toMap(PaymentStrategy::getType, Function.identity()));
    }

    public PaymentResult pay(String type, PaymentRequest request) {
        PaymentStrategy strategy = strategyMap.get(type);
        if (strategy == null) {
            throw new UnsupportedPaymentTypeException(type);
        }
        return strategy.pay(request);
    }
}
```

新增支付方式？只要加一個新的 `@Component` 實作類別，**完全不需要改動現有程式碼**。

### 4.2 Template Method — 框架的骨架

**核心思想**：在父類別定義演算法骨架，子類別只覆寫特定步驟。

#### 實際場景：不同格式的報表匯出

```java
public abstract class ReportExporter<T> {

    // 模板方法：定義匯出流程的骨架
    public final byte[] export(ReportQuery query) {
        List<T> data = fetchData(query);
        List<T> processed = processData(data);
        byte[] output = render(processed, query.getTitle());
        logExport(query);
        return output;
    }

    protected abstract List<T> fetchData(ReportQuery query);

    protected List<T> processData(List<T> data) {
        return data; // 預設不處理，子類別可覆寫
    }

    protected abstract byte[] render(List<T> data, String title);

    private void logExport(ReportQuery query) {
        log.info("匯出報表：type={}, query={}", getFormat(), query);
    }

    protected abstract String getFormat();
}

@Component
public class ExcelReportExporter extends ReportExporter<SalesRecord> {

    @Override
    protected List<SalesRecord> fetchData(ReportQuery query) {
        return salesRepository.findByDateRange(query.getStart(), query.getEnd());
    }

    @Override
    protected byte[] render(List<SalesRecord> data, String title) {
        // 使用 Apache POI 產生 Excel
        // ...
    }

    @Override
    protected String getFormat() { return "EXCEL"; }
}
```

Spring 自身大量使用這個模式：`JdbcTemplate` 把「取得連線 → 執行 SQL → 處理結果 → 釋放連線」的骨架固定下來，你只需要關注 SQL 和結果映射。

### 4.3 Observer — 事件驅動

**核心思想**：當某件事發生時，自動通知所有關注者。

#### Spring ApplicationEvent + @EventListener

```java
// 定義事件
public record OrderCreatedEvent(Long orderId, Long customerId, BigDecimal amount) {}

// 發布事件
@Service
@RequiredArgsConstructor
public class OrderService {

    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public Order createOrder(OrderRequest request) {
        Order order = orderRepository.save(toEntity(request));

        // 發布事件，不需要知道誰在監聽
        eventPublisher.publishEvent(
            new OrderCreatedEvent(order.getId(), order.getCustomerId(), order.getTotalAmount()));

        return order;
    }
}

// 監聽者 1：扣減庫存
@Component
public class InventoryListener {

    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        inventoryService.deduct(event.orderId());
    }
}

// 監聽者 2：發送通知
@Component
public class NotificationListener {

    @Async
    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        notificationService.sendOrderConfirmation(event.customerId(), event.orderId());
    }
}

// 監聽者 3：記錄日誌
@Component
public class AuditLogListener {

    @EventListener
    public void onOrderCreated(OrderCreatedEvent event) {
        auditLogService.log("ORDER_CREATED", event.orderId(), event.amount());
    }
}
```

**優勢**：`OrderService` 完全不知道有幾個監聽者，要新增一個「訂單建立後同步到 ERP」的需求，只需要加一個新的 `@EventListener`，零修改現有程式碼。

### 4.4 Chain of Responsibility — 處理鏈

**核心思想**：把請求沿著處理鏈傳遞，每個節點決定自己要不要處理、要不要往下傳。

#### Spring Security FilterChain

你每天用的 Spring Security 就是這個模式：

```
HTTP 請求 → CorsFilter → CsrfFilter → AuthenticationFilter → AuthorizationFilter → Controller
```

#### 實際場景：多層驗證

```java
public interface ValidationHandler {
    ValidationResult validate(OrderRequest request);
    int getOrder(); // 決定執行順序
}

@Component
public class FormatValidationHandler implements ValidationHandler {

    @Override
    public ValidationResult validate(OrderRequest request) {
        if (request.getCustomerId() == null) {
            return ValidationResult.fail("客戶 ID 不可為空");
        }
        if (request.getItems() == null || request.getItems().isEmpty()) {
            return ValidationResult.fail("訂單項目不可為空");
        }
        return ValidationResult.ok();
    }

    @Override
    public int getOrder() { return 1; }
}

@Component
public class PermissionValidationHandler implements ValidationHandler {

    @Override
    public ValidationResult validate(OrderRequest request) {
        if (!currentUser.hasPermission("ORDER_CREATE")) {
            return ValidationResult.fail("無建立訂單權限");
        }
        return ValidationResult.ok();
    }

    @Override
    public int getOrder() { return 2; }
}

@Component
public class BusinessRuleValidationHandler implements ValidationHandler {

    @Override
    public ValidationResult validate(OrderRequest request) {
        if (request.getTotalAmount().compareTo(creditLimit) > 0) {
            return ValidationResult.fail("超過信用額度");
        }
        return ValidationResult.ok();
    }

    @Override
    public int getOrder() { return 3; }
}

// 執行驗證鏈
@Service
public class OrderValidationService {

    private final List<ValidationHandler> handlers;

    public OrderValidationService(List<ValidationHandler> handlers) {
        this.handlers = handlers.stream()
                .sorted(Comparator.comparingInt(ValidationHandler::getOrder))
                .toList();
    }

    public ValidationResult validate(OrderRequest request) {
        for (ValidationHandler handler : handlers) {
            ValidationResult result = handler.validate(request);
            if (!result.isValid()) {
                return result; // 驗證失敗，中斷鏈
            }
        }
        return ValidationResult.ok();
    }
}
```

## 5、何時不該用設計模式

### YAGNI：不需要的彈性不要加

**You Aren't Gonna Need It。** 如果系統目前只有兩種支付方式，而且產品經理說「短期不會加」，那直接用 `if-else` 反而更清楚。等到第三種出現時再重構也不遲。

### 反模式：為了用模式而用模式

常見的過度設計：

- 只有一種實作，卻硬要抽介面 + 工廠
- 三行邏輯包成 Strategy，類別數量暴增
- 所有事件都用 Observer，結果除錯時追不到呼叫鏈

### 決策表：什麼場景用什麼模式

| 場景特徵 | 建議模式 | 不建議使用的情況 |
|---------|---------|----------------|
| 同一介面有 3+ 種實作，執行時期動態切換 | Strategy | 只有 1-2 種且不會增加 |
| 物件建構參數多、可選組合複雜 | Builder | 參數 ≤ 3 個且都是必填 |
| 需要在不改原始碼的情況下增加功能 | Decorator | 功能固定，不需要動態組合 |
| 一個動作需要觸發多個不相關的後續處理 | Observer | 後續處理只有一個且邏輯簡單 |
| 請求需經過多個獨立的處理步驟 | Chain of Responsibility | 步驟固定且只有 2 步 |
| 多個子類別的流程骨架相同，細節不同 | Template Method | 只有一個子類別 |
| 需要攔截方法呼叫，加入橫切邏輯 | Proxy（AOP） | 直接用 Spring `@Aspect` 即可 |

## 6、小結

| 模式 | Spring 中的實例 | 典型場景 | 濫用風險 |
|------|----------------|---------|---------|
| Factory Method | `BeanFactory`、`FactoryBean` | 根據類型建立不同處理器 | 只有一種實作時多此一舉 |
| Builder | `UriComponentsBuilder`、Lombok `@Builder` | 複雜查詢條件、DTO 建構 | 簡單物件不需要 |
| Singleton | Spring Bean 預設 Scope | Service、Repository | 可變狀態導致執行緒安全問題 |
| Decorator | `HttpServletRequestWrapper`、I/O Stream | 快取裝飾、日誌裝飾 | 層數太多難以除錯 |
| Proxy | Spring AOP、`@Transactional` | 交易管理、權限檢查、日誌 | 同類別內部呼叫不生效 |
| Strategy | 依賴注入 `Map<String, T>` | 多種支付、多種匯出格式 | 只有兩種情況時過度抽象 |
| Template Method | `JdbcTemplate`、`RestTemplate` | 報表匯出、資料匯入 | 繼承層次過深 |
| Observer | `ApplicationEvent` + `@EventListener` | 訂單建立後的連鎖處理 | 事件鏈太長難以追蹤 |
| Chain of Responsibility | Spring Security `FilterChain` | 多層驗證、請求過濾 | 鏈太長影響效能 |
