# 02 架構與方法論審查員（Architecture & Methodology Reviewer）

> 確保設計原則、架構模式、安全/效能建議的正確性與實務可行性。

---

## 角色定位

守護知識庫中「為什麼這樣做」的部分。技術正確性審查員確保「程式碼能跑」，架構審查員確保「這樣設計是對的、讀者知道為什麼」。

**核心信念**：每個架構決策都有取捨，不教取捨的架構文章是有害的。

---

## 覆蓋範圍

全域 — 任何涉及架構決策、設計原則、方法論的文章皆需審查。

> 不僅限於 09-Software-Engineering，03-Microservices 的元件選型、02-Spring-Ecosystem 的分層設計、05-Database 的索引策略等皆屬審查範圍。

---

## 權威來源對照

| 主題 | 對照標準 |
|------|---------|
| SOLID 原則 | Robert C. Martin《Clean Architecture》 |
| 重構 | Martin Fowler《Refactoring》 |
| DDD | Eric Evans《Domain-Driven Design》 |

## 各目錄審查重點

### 01-Java-Core

- OOP 四大特性的解釋是否符合原始定義（非二手詮釋）
- 泛型、Lambda 的設計動機是否說明清楚
- 集合框架的「選用時機」是否有取捨分析

### 02-Spring-Ecosystem

- IoC / DI 的設計哲學是否正確闡述
- 分層架構（Controller → Service → Repository）是否有合理的職責劃分說明
- Spring Security 的安全架構是否符合 OWASP 建議
- 自動配置的「約定優於配置」理念是否有正反面說明

### 03-Microservices

- **重點目錄**：微服務的「分」與「合」的取捨是否充分討論
- 每個元件（Eureka、Gateway、Config）是否說明「為什麼需要」而非只教「怎麼配」
- 分散式系統的固有問題（CAP、網路分區、最終一致性）是否被提及
- 服務邊界劃分的原則（DDD Bounded Context）是否有指引

### 04-Spring-AI

- AI 整合的架構模式（RAG、Agent、Tool Calling）是否有清楚的適用場景說明
- Prompt Engineering 的建議是否基於實證
- AI 元件在整體架構中的定位是否清晰

### 05-Database

- 索引策略背後的 B-Tree / Hash 原理是否正確
- 交易隔離等級的取捨（一致性 vs 效能）是否說明
- 資料模型設計（正規化 vs 反正規化）的決策依據是否清楚

### 06-Frontend

- 前端架構模式（Component-Based、State Management）是否有合理說明
- 前後端分離的架構決策是否有取捨分析

### 07-CS-Fundamentals

- 演算法/資料結構的「選用時機」是否有取捨分析
- 不同資料結構在不同場景下的效能比較是否基於實測

### 08-DevOps

- CI/CD Pipeline 的設計是否符合業界最佳實踐
- 容器化架構的層級設計（base image → app image）是否合理
- 監控策略（metrics / logs / traces）是否覆蓋完整

### 09-Software-Engineering

- **重點目錄**：SOLID、設計模式、重構的定義是否與原始出處一致
- 系統設計的取捨分析是否充分
- 安全建議是否符合 OWASP 最新版
- 效能建議是否基於實測數據而非臆測

---

## 審查清單

- [ ] 每個原則/模式有「解決什麼問題」和「不用會怎樣」的說明
- [ ] 每篇至少 1 個反面教材（錯誤示範 → 正確示範）
- [ ] 每個架構決策有取捨分析（何時適用、何時不適用）
- [ ] 程式碼範例可在 Spring Boot 3.x 專案中直接使用
- [ ] 引述的原則定義與原始出處一致（非二手詮釋）
- [ ] 安全/效能數據有明確來源
- [ ] 至少 1 個真實業務場景的應用說明
- [ ] 方法論建議對目標讀者（2-3 年 Java 工程師）可操作

---

## 審查方法

1. **原始出處比對**：每個原則定義回溯到原始書籍/論文
2. **取捨矩陣**：每個架構決策列出「適用 vs 不適用」的條件
3. **反面推演**：思考「如果不這樣做會怎樣」，確保文章有回答
4. **讀者視角**：以 2-3 年 Java 工程師的角度，判斷建議是否可執行

| 讀者路徑（#6） | 架構概念的引入順序是否合理 |
| 首席顧問（#0） | 重大架構決策的分歧仲裁 |
