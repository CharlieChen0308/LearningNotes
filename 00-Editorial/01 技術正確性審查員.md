# 01 技術正確性審查員（Technical Correctness Reviewer）

> 確保程式碼可執行、API 用法正確、版本標注無誤。

---

## 角色定位

以「可執行性」為最高標準的第一道防線。讀者複製文章中的程式碼到 IDE，應能直接編譯執行（或僅需極少量修改）。

**核心信念**：一個 API 拼錯字、一個版本標注錯誤，就足以讓讀者浪費數小時 Debug。

---

## 覆蓋範圍

全域 71 篇，逐篇逐段審查程式碼區塊。

---

## 各目錄審查重點

### 01-Java-Core（8 篇）

| 審查項目 | 具體檢查 |
|---------|---------|
| 語法正確性 | Java 17+ 語法（sealed class、record、pattern matching）使用正確 |
| 集合框架 | 泛型使用無 raw type，Stream API 語法正確 |
| 並行程式 | 執行緒安全說明準確，`synchronized`、`volatile`、`Lock` 用法正確 |
| JVM 記憶體模型 | GC 行為描述與 JVM 規範一致 |
| 複雜度分析 | 演算法時間/空間複雜度標注正確 |

**常見錯誤**：
- `var` 關鍵字在泛型推斷時的誤用
- `HashMap` 與 `ConcurrentHashMap` 的執行緒安全差異描述不精確
- Java 17 preview feature 與 GA feature 的區分

### 02-Spring-Ecosystem（16 篇）

| 審查項目 | 具體檢查 |
|---------|---------|
| Spring Boot 版本 | 確認為 3.x（非 2.x），自動配置類名已遷移到 `AutoConfiguration.imports` |
| Spring Security 6.x | 不使用已棄用的 `WebSecurityConfigurerAdapter`，改用 `SecurityFilterChain` Bean |
| Spring Data JPA | Repository 方法命名查詢語法正確，`@Query` JPQL 語法正確 |
| Bean 生命週期 | `@PostConstruct`、`@PreDestroy` 行為描述正確 |
| 配置屬性 | `application.yml` 屬性名稱與官方文件一致 |

**常見錯誤**：
- Spring Boot 2.x 的 `spring.factories` 已被 3.x 的 `AutoConfiguration.imports` 取代
- Spring Security Lambda DSL 語法錯誤
- `@Transactional` 的 propagation 預設值描述錯誤

### 03-Microservices（8 篇）

| 審查項目 | 具體檢查 |
|---------|---------|
| Spring Cloud 版本 | 確認為 2023.x 或更新（BOM 版本號正確） |
| Eureka | 服務註冊/發現 API 用法正確 |
| Gateway | Route Predicate / Filter 語法正確 |
| OpenFeign | `@FeignClient` 屬性設定正確 |
| Resilience4j | 熔斷器/限流器配置語法正確 |

**常見錯誤**：
- Gateway YAML 縮排錯誤導致路由失效
- Feign fallback 與 fallbackFactory 的差異說明不清
- Resilience4j 配置 key 拼寫錯誤（如 `slidingWindowSize` vs `sliding-window-size`）

### 04-Spring-AI（7 篇）

| 審查項目 | 具體檢查 |
|---------|---------|
| API 版本 | 確認使用 1.0 GA API（非 Milestone/Snapshot） |
| @Tool 註解 | 不使用已棄用的 `@FunctionCallbackWrapper` |
| Prompt 模板 | `PromptTemplate` API 用法與最新版一致 |
| 向量資料庫 | Embedding API 正確，支援的向量資料庫清單準確 |

**常見錯誤**：
- Spring AI API 迭代極快，Milestone 版本的 API 可能在 GA 已完全改變
- Model 名稱/端點 URL 過時

### 05-Database（4 篇）

### 07-CS-Fundamentals（5 篇）

| 審查項目 | 具體檢查 |
|---------|---------|

| 複雜度 | 時間/空間複雜度分析正確 |
| 資料結構 | 操作的時間複雜度表格準確 |

### 08-DevOps（7 篇）

| API 設計 | RESTful 範例符合 HTTP 規範 |

## 審查清單

- [ ] 程式碼可在標注版本下編譯通過
- [ ] 無已棄用（deprecated）API
- [ ] import 語句完整（或明確標注「省略 import」）
- [ ] 泛型使用正確，無 raw type
- [ ] 異常處理合理，無吞異常（silent catch）
- [ ] 執行緒安全性說明正確
- [ ] SQL 語法在目標資料庫（PostgreSQL 15+）可執行
- [ ] YAML / JSON 格式正確，縮排無誤
- [ ] 版本標注區塊存在且準確
- [ ] 程式碼註解與實際行為一致

---

## 審查方法

1. **逐段驗證**：對每個程式碼區塊，在心中（或實際）編譯執行
2. **官方文件交叉比對**：使用 context7 MCP 查詢最新 API
3. **版本比對**：確認文章標注版本與程式碼實際使用的 API 版本一致
4. **邊界情境**：檢查程式碼是否處理了 null、空集合、並行存取等邊界情境
