# 驗收標準（Acceptance Criteria）

> 所有審查角色共用。將 Checklist 的布林值（勾/不勾）轉化為可量化、可重複驗證的具體標準。

---

## 為什麼需要驗收標準

Checklist 回答「查了沒有」，驗收標準回答「查到什麼程度才算過」。沒有驗收標準，同一篇文章嚴格審和寬鬆審結果截然不同。

---

## 第一輪 — 技術正確性（#1）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 程式碼可在標注版本下編譯通過 | 逐段心智編譯 + 使用 context7 交叉比對 API 簽名。若程式碼涉及多個類別協作，需確認整體可組裝 |
| 無已棄用 API | 使用 context7 查詢每個核心 API 的棄用狀態。已棄用 = 官方文件標記 `@Deprecated` 或 Release Notes 明確聲明 |
| import 語句完整 | 完整列出所有非 `java.lang` 的 import，或在程式碼區塊上方標注 `// 省略 import` |
| 泛型使用正確 | 無 raw type（如 `List` 而非 `List<String>`），無 unchecked cast 警告 |
| 異常處理合理 | 無空 catch 區塊，或空 catch 有明確註解說明原因 |
| 版本標注準確 | `> **版本**：` 區塊的每個版本號可在 Maven Central / npm 查到對應的 GA 版本 |
| YAML/JSON 格式正確 | 縮排一致（YAML 用 2 空格），屬性名稱與官方文件一致 |

## 第一輪 — 架構與方法論（#2）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 原則定義與原始出處一致 | 可回溯到權威來源對照表中的對應書籍/規範，非二手部落格轉述 |
| 每篇至少 1 個反面教材 | 有明確的「錯誤示範」程式碼/架構圖，且有「為什麼錯」的解釋，再接「正確示範」 |
| 每個架構決策有取捨分析 | 至少包含：(1) 何時適用 (2) 何時不適用 (3) 替代方案是什麼。三者缺一不可 |
| 方法論建議可操作 | 以 2-3 年 Java 工程師為判斷基準：讀完是否知道「明天上班怎麼用」。純理論無操作指引 = 不通過 |
| 安全/效能數據有來源 | 數據需標明出處（官方 benchmark、OWASP 文件、JVM 規範）。「通常比較快」這類無依據的描述 = 不通過 |

## 第一輪 — 生產實戰（#3）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 業界採用狀態說明 | 每個核心元件必須標明：活躍 / 維護模式 / 已棄用 / 新興。判斷依據：官方公告 + GitHub 最近 6 個月的 commit 頻率 |
| 替代方案比較 | 至少提及 1 個主流替代方案，且有簡要的比較（≥ 3 個比較維度：適用場景、學習曲線、社群活躍度等） |
| 生產環境注意事項 | 教學簡化範例必須有 `> 此為教學簡化範例，生產環境需額外考慮：` 的標注，後接至少 2 點生產考量 |
| 無純 localhost 範例 | 若範例僅適用本機（如 `localhost:8080` 硬編碼），必須標注「教學用途」且說明生產環境的配置方式 |
| 敏感配置不外洩 | 範例中的密碼、API Key 使用 `${ENV_VAR}` 或 `*****` 佔位，不出現實際值 |

## 第二輪 — 內容結構（#4）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 文章骨架完整 | 必須包含：版本標注、概述/是什麼、實作/怎麼做、小結、延伸閱讀。缺任一 = 不通過 |
| 標題層級正確 | H1 僅 1 個，H2 為主章節（3-8 個），不超過 H4。實際計數驗證 |
| 概念引入順序 | 遵循「是什麼 → 為什麼 → 怎麼做 → 注意什麼」。若先出現實作再解釋概念 = 不通過 |
| 程式碼區塊規範 | 有語言標注、單區塊 ≤ 50 行、不連續出現兩個區塊（中間有說明文字） |
| 無 ASCII art | 掃描全文，不得出現 `┌─┐│└┘├┤` 等框線字元。有 = 不通過 |

---

## 第二輪 — 術語與一致性（#5）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 術語統一 | 全文搜索術語對照表「禁用譯法」欄位的所有詞彙，出現任一 = 不通過 |
| 英文前後空格 | 中文字元與英文/數字之間有半形空格。例外：反引號內的程式碼不需要 |
| 無簡體殘留 | 搜索常見簡體字元（对、与、进、关、这、们、过、动、发、开）。出現任一 = 不通過 |
| 全形標點 | 中文語境使用全形標點（，。；：「」（）），程式碼區塊內不受限 |
| 連結有效性 | 所有 `[xxx](xxx.md)` 連結的目標檔案存在。可用腳本批次驗證 |
| 跨目錄引用雙向 | A 引用 B 時，B 的延伸閱讀也應引用 A。單向引用需標記為待補 |
| 概念定義一致 | 同一概念（如 DI、交易隔離）在不同篇章的定義不矛盾。抽查至少 3 個跨篇概念 |

---

## 第三輪 — 讀者路徑（#6）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 前置知識已覆蓋 | 文章中使用的每個未解釋概念，必須在 README 建議路線的前序文章中已教授。逐一比對 |
| 同目錄難度遞增 | 以「引入的新概念數量」和「程式碼複雜度」為指標，後篇 ≥ 前篇 |
| 閱讀時間合理 | 以字數估算：中文正文約 300 字/分鐘，程式碼約 150 字/分鐘（需理解）。全篇控制在 15-25 分鐘 = **4500-7500 中文字**（含程式碼折算）。超過 10000 字建議拆篇 |
| 無孤島文章 | 每篇至少被 README 或另一篇文章引用。完全孤立 = 不通過 |
| 實務範例 | 每篇至少 1 個「實際開發中會這樣用」的範例，純教科書式但實務中不會用的範例不算 |

---

## 第三輪 — 時效性（#7）

| Checklist 項目 | 驗收標準 |
|---------------|---------|
| 版本為當前穩定版或前一版 | 對照 `07 時效性審查員.md` 的版本基準線表。低於警戒線 = 不通過 |
| 無維護模式元件作為首選 | 元件狀態對照基準線表。維護模式元件可教學但必須 (1) 標注狀態 (2) 提及替代方案 |
| 無 @Deprecated API | 使用 context7 查詢。已標記 @Deprecated 的 API 不得作為正規教學用法 |
| 版本基準線本身的時效 | 每半年（1 月 / 7 月）由時效性審查員更新基準線表中的版本號 |

---

## 驗收標準的維護

| 項目 | 說明 |
|------|------|
| **誰維護** | 各角色負責自己對應的驗收標準，總編輯統籌 |
| **更新時機** | 季度回顧時，依據實際審查經驗調整（見缺陷 3 — 回顧改善機制） |
| **新增標準** | 任何角色可提議新增，經合議確認後加入 |
| **爭議解決** | 對驗收標準本身有分歧 → 總編輯仲裁 → 無法解決 → 首席顧問裁定 |
