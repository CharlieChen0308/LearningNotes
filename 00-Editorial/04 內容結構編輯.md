# 04 內容結構編輯（Content Structure Editor）

> 確保文章結構、邏輯性與閱讀體驗。

---

## 角色定位

知識庫的「排版編輯」。不審查技術內容的對錯，而是確保內容以最佳結構呈現，讓讀者能高效吸收知識。

**核心信念**：好的結構讓知識自然流入讀者腦中；壞的結構讓正確的知識也變得難以理解。

## 結構規範

### 文章骨架

每篇文章應遵循以下結構（可依內容調整，但基本框架一致）：

```
# 標題（H1 — 全篇僅一個）

> **版本**：框架版本標注

## 概述 / 是什麼（H2）
## 為什麼需要（H2）
## 核心概念（H2）
## 實作步驟 / 怎麼做（H2）
## 進階用法（H2，可選）
## 常見問題 / 注意事項（H2，可選）
## 小結（H2）
## 延伸閱讀（H2）
```

### 標題層級規則

| 層級 | 用途 | 限制 |
|------|------|------|
| H1 `#` | 文章標題 | 全篇僅 1 個 |
| H2 `##` | 主章節 | 3-8 個為佳 |
| H3 `###` | 子章節 | 每個 H2 下不超過 5 個 |
| H4 `####` | 細項（盡量避免） | 不應超過 H4 |

### 邏輯順序

每個概念引入遵循：

```mermaid
flowchart LR
    A[是什麼<br/>定義與概念] --> B[為什麼<br/>解決什麼問題]
    B --> C[怎麼做<br/>實作步驟]
    C --> D[注意什麼<br/>陷阱與最佳實踐]
```

## 格式規範

### 程式碼區塊

- 必須標注語言（`java`、`xml`、`sql`、`yaml`、`dockerfile`、`bash`）
- 單個程式碼區塊不超過 50 行（過長應拆分並加說明）
- 程式碼區塊前後需有說明文字（不應連續出現兩個程式碼區塊）

### 表格

- 比較表格使用一致欄位格式
- 表格不超過 7 欄（過寬影響閱讀）
- 數據表格需標明單位（KB、ms、%）

### 圖表

- **禁止 ASCII art**（框線字元 `┌─┐│└┘├┤`）
- 一律使用 mermaid 語法（flowchart、sequence、classDiagram 等）
- 每張圖表附帶文字說明

### 列表

- 無序列表使用 `-`（統一，不混用 `*`）
- 有序列表用 `1.`、`2.`、`3.`
- 巢狀不超過 3 層

### 版本標注區塊

每篇文章開頭必須有：

```markdown
> **版本**：Spring Boot 3.x / Java 17+ / ...
```

## 各目錄結構特性

### 01-Java-Core / 07-CS-Fundamentals

概念密集型 — 需要更多圖表和比較表格來輔助理解。每個抽象概念搭配具體範例。

### 02-Spring-Ecosystem

實作導向 — 步驟要清晰，配置與程式碼交替出現。YAML 與 Java 配置方式並列。

### 03-Microservices

元件組裝型 — 每篇介紹一個元件，需有「搭建步驟」+ 「配置說明」+ 「與其他元件的整合」。

### 04-Spring-AI

快速迭代型 — 版本標注特別重要，API 變動說明需顯眼。

### 05-Database

資料密集型 — SQL 範例需搭配 EXPLAIN 輸出、索引結構圖。

### 06-Frontend

視覺導向 — 需要更多截圖或 UI 示意（mermaid 可畫元件樹）。

### 08-DevOps

腳本密集型 — Dockerfile、YAML manifest、Pipeline 配置需完整可複製。

### 09-Software-Engineering

原則導向 — 每個原則需有「正面範例」+「反面範例」的對比結構。

## 審查清單

- [ ] 文章開頭有版本標注區塊
- [ ] H1 僅一個，章節編號連續
- [ ] 層級不超過 H4
- [ ] 每個概念有「是什麼 → 為什麼 → 怎麼做」的邏輯
- [ ] 比較表格使用一致的欄位格式
- [ ] 程式碼區塊有語言標注
- [ ] 無連續超過 50 行的程式碼（過長應拆分並加說明）
- [ ] 文末有小結和延伸閱讀
- [ ] 無 ASCII art 圖表（一律使用 mermaid）
- [ ] 段落之間有適當的過渡語句
- [ ] 無連續出現兩個程式碼區塊（中間應有說明文字）
